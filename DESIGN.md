# Pet Platform Design

## 1. 设计目标

平台需要同时体现宠物服务的温度和交易系统的效率。界面应清晰、可信、易扫描，让用户快速完成登录、查看工作台、寻找服务、购买用品和查询医院。

关键词：

- 可信：认证、审核状态、评分、订单状态和审计信息清晰可见。
- 高效：核心操作路径短，列表信息密度适中。
- 温暖：图片和文案体现宠物照护场景，但不牺牲业务可读性。
- 一致：网页端、后续 App、小程序共用同一套视觉语言和术语。

## 2. 视觉规范

### 2.1 色彩

- 主色：`#2F80ED`，用于主按钮、链接和关键状态。
- 成功色：`#27AE60`，用于可预约、成功、健康状态。
- 警示色：`#F2994A`，用于待处理、库存预警、提醒。
- 错误色：`#EB5757`，用于失败、取消、风险提示。
- 文本主色：`#1F2933`。
- 文本次色：`#64748B`。
- 背景色：`#F7F9FC`。
- 边框色：`#E5EAF0`。

### 2.2 字体

- Web 使用 `system-ui, -apple-system, BlinkMacSystemFont, "Segoe UI", sans-serif`。
- 中文正文建议 14-16px。
- 工作台、卡片和表格标题保持克制，避免营销页式超大字号。
- 字间距保持 `0`，不使用负字距。

### 2.3 图标与图片

- 工具按钮优先使用常见线性图标。
- 宠物服务、商品和医院模块使用真实或清晰的业务图片。
- 图片必须有 `alt` 文本。
- 图标按钮需要 hover/焦点提示。

## 3. 布局规范

### 3.1 网页端

- 当前 MVP 使用静态 HTML/CSS/JS 实现。
- 首屏直接进入可用应用体验，不做纯营销落地页。
- 桌面端采用顶部导航 + 角色工作台布局。
- 内容最大宽度建议 1200-1440px。
- 卡片用于单个业务对象，如服务、商品、医院、指标。
- 避免卡片嵌套卡片，页面区块使用全宽 band 或无框布局。

### 3.2 App

- 建议底部 Tab：首页、服务、商城、医院、我的。
- 下单流程使用分步页面，避免单页信息过载。
- 订单状态使用时间线展示。
- 图片上传、定位、支付和推送使用平台能力适配层。

### 3.3 小程序

- 优先覆盖高频操作：找服务、买商品、查医院、看订单。
- 页面首屏请求数量要少。
- 登录、定位、支付和订阅消息封装为平台适配能力。

## 4. 核心页面

### 4.1 登录注册

- 支持登录和注册切换。
- 手机号为主凭证，先完成网页图形验证码，再获取短信验证码。
- 注册时明确选择角色。
- 登录失败时展示真实错误，不默认伪造成功。
- Demo 模式只能由用户显式勾选启用。
- 提交期间按钮 disabled，并显示 loading 文案。

### 4.2 角色工作台

- 普通用户：宠物档案、待服务订单、购物订单、附近医院、健康提醒。
- 店家：今日订单、待发货、库存预警、售后申请、商品表现。
- 医院：待确认预约、今日到院、资质状态、健康内容。
- 管理员：待审核、投诉工单、退款申请、平台 GMV、审计日志。

### 4.3 服务列表

- 支持按服务类型、时间、距离、评分和价格筛选。
- 服务卡片包含姓名、认证、距离、评分、完成单量和起步价。
- 空状态提供重试或联系客服入口。

### 4.4 商城

- 商品卡片展示图片、标题、分类、价格、库存和状态。
- 详情页应突出适用品种、成分、规格、配送和售后。
- 店家侧要突出库存预警、上下架和订单状态。

### 4.5 医院

- 医院卡片展示名称、地址、电话、营业状态和评分。
- 医院工作台展示预约意向和资料审核状态。
- 后续可扩展地图列表切换。

## 5. 交互规范

- 主按钮用于页面最重要动作。
- 危险操作需要二次确认。
- 表单错误展示在字段附近。
- 异步操作必须有 loading、empty、error 状态。
- 网络失败提供重试。
- 金额、时间、订单状态使用统一格式化。

## 6. 可访问性

- 文本与背景对比度满足 WCAG AA 建议。
- 移动端点击热区不小于 44px。
- 表单字段必须有明确 label。
- 键盘可以访问主要操作。
- 焦点状态清晰可见。
