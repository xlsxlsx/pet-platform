# Pet Platform PRD

## 1. 产品概述

Pet Platform 是面向家养宠物生活服务的一体化平台，覆盖网页端、后续 App 和小程序。平台围绕宠物主的日常照护、上门服务、宠物用品购买、宠物医院查询、健康档案和订单履约，连接普通用户、宠物用品店家、宠物医院和平台管理员。

当前阶段目标是先完成可本地运行和演示的网页端 MVP：

- 四类角色统一登录注册。
- 普通用户查看宠物、服务、商品、医院和订单概览。
- 店家查看商品、库存、商城订单和经营指标。
- 宠物医院查看医院资料、预约和资质状态。
- 管理员查看待审核事项、投诉工单、平台数据和审计信息。

## 2. 用户角色

| 角色 | 编码 | 主要目标 |
| --- | --- | --- |
| 普通用户 | `CUSTOMER` | 管理宠物档案，预约上门服务，购买猫粮狗粮等用品，寻找宠物医院 |
| 宠物用品店家 | `MERCHANT` | 管理店铺资料、商品、库存、商城订单和售后 |
| 宠物医院 | `HOSPITAL` | 管理医院资料、执业资质、预约意向、医生和医疗服务内容 |
| 管理员 | `ADMIN` | 管理用户、审核店家和医院、处理投诉、查看平台数据和审计日志 |

## 3. 核心功能

### 3.1 账号与权限

- 支持手机号 + 网页图形验证码 + 短信验证码注册登录。
- 登录和注册均以手机号验证码为主，不在前端或 API 暴露密码字段。
- 注册时选择角色：普通用户、店家、医院、管理员。
- 管理员注册默认受配置开关保护，避免公开环境随意创建管理员。
- 使用 RBAC 模型管理角色与权限点。
- 不同角色登录后进入对应工作台。

### 3.2 普通用户

- 宠物档案：姓名、物种、品种、性别、生日、健康备注。
- 地址簿：联系人、电话、省市区、详细地址。
- 上门服务：查看服务人员、服务类型、价格、评分和历史订单。
- 商城：查看商品、价格、库存和订单状态。
- 医院：查询附近宠物医院、营业状态、电话和预约入口。
- 工单与评价：投诉、售后、服务评价。

### 3.3 店家

- 店铺资料和审核状态。
- 商品管理：名称、分类、价格、库存、上下架状态。
- 商城订单：待发货、配送中、售后、退款。
- 经营数据：订单量、GMV、库存预警、售后率。

### 3.4 宠物医院

- 医院资料和执业许可证。
- 医院列表展示：地址、电话、营业状态、评分。
- 预约意向处理：待确认、到院、完成、取消。
- 后续扩展医生、科室、体检、疫苗和驱虫提醒。

### 3.5 管理员

- 用户和角色管理。
- 店家、医院、服务人员审核。
- 投诉、退款、风控和黑名单。
- 平台运营配置：分类、公告、优惠、推荐位。
- 审计日志和数据看板。

## 4. MVP 范围

已落地或优先落地：

- FastAPI 后端基础结构。
- MySQL 初始化脚本和四角色测试账号。
- 四角色登录注册接口。
- 四角色工作台接口。
- 商品和医院目录查询接口。
- 静态网页端登录注册、角色切换、工作台、服务、商城和医院展示。
- 本地数据库初始化、可见终端启动说明和测试账号文档。

暂缓：

- 真实支付、退款和物流接入。
- 真实短信服务商、微信/支付宝登录。
- App 和小程序工程化实现。
- 实时地图、在线问诊、智能推荐。
- 完整管理后台和复杂权限配置界面。

## 5. 关键流程

### 5.1 注册登录

1. 用户选择登录或注册。
2. 用户填写手机号和网页图形验证码，后端校验图形验证码后发放短信验证码。
3. 注册时填写手机号、短信验证码、昵称和角色。
4. 后端校验短信验证码，创建用户、绑定角色并生成对应 profile。
5. 登录成功后返回用户信息和角色列表。
6. 前端根据角色加载对应工作台。

### 5.2 普通用户找服务

1. 用户进入工作台查看宠物档案和服务推荐。
2. 浏览上门喂养、遛狗、寄养等服务。
3. 查看服务人员评分、服务范围和价格。
4. 下单、支付、履约、报告、评价在后续版本补齐。

### 5.3 店家处理订单

1. 店家登录工作台。
2. 查看今日订单、待发货和库存预警。
3. 维护商品和库存。
4. 后续接入发货、物流和售后流程。

### 5.4 医院处理预约

1. 医院登录工作台。
2. 查看预约意向和到院安排。
3. 维护医院资料、电话、地址和营业状态。
4. 后续扩展医生排班和医疗服务套餐。

## 6. 非功能需求

- 数据库使用 MySQL 8.0、InnoDB、utf8mb4。
- 金额统一使用整数分，避免浮点误差。
- 敏感信息在日志和后台展示时需要脱敏。
- 高风险操作写入审计日志。
- 后端按清洁架构分层，保持高内聚、低耦合。
- 接口统一使用 `/api/v1` 前缀。
- 本地环境必须能通过可见终端命令启动，并能通过冒烟脚本验证核心接口。
